IBOR Platform Requirements
==========================
Objective: Build an IBOR platform that should:
1. Display transactions by an asset class
2. Display Positions on those transaction (long / short)
3. Display cash ladder (cash inflow/outflow) of next 7 days
4. User should be able to select the position by Portfolio / Account


=============================================================
PRODUCT VISION & ROADMAP
=============================================================

Platform Name: TBD (domain discussion deferred — to revisit after Phase 1 is live)

Platform Positioning:
  Most firms buy Security Master, IBOR, and Backtesting as three separate $1M+ enterprise
  systems. This platform delivers all three as one integrated, AI-native product.
  Core philosophy: numbers/facts come from SQL, reasoning/narrative come from AI.

Data Hierarchy (agreed model):
  Fund (top level)
   └── Portfolio  (investment strategy / mandate — the PM's view)
         └── Account  (where assets are custodied / settled — the ops view)

  - One Portfolio → Many Accounts (e.g. prime brokerage, custody, cash account)
  - dim_account and dim_account_portfolio already exist in schema with SCD2
  - dim_account_portfolio is a many-to-many mapping (flexible, supports edge cases)
  - Current gap: JodiPositionsRepository does not filter by account_code yet
  - Fix deferred to Phase 2 — Phase 1 demo uses single portfolio, no account filter needed

  [P1-FIX-8]  Account-level filtering missing — Spring Boot queries ignore account_code
    Problem:   JodiPositionsRepository filters by portfolio_vid only. The schema has
               dim_account, dim_account_portfolio (many-to-many), and fact_trade.account_vid
               — the data hierarchy is Portfolio -> Account -> Trade — but no API endpoint

               This blocks multi-tenant use where a PM manages multiple accounts

    Fix:       (1) Add optional accountCode param to GET /api/positions in Spring Boot
               (2) Add account CTE to JodiPositionsRepository SQL that joins through
                   dim_account_portfolio when accountCode is provided
               (3) Expose accountCode in the AI tool definition (plan_query ibor_calls args)
               (4) Pass it through _dispatch_ibor in llm_service.py
    Files:     ibor-server/src/main/java/com/kmakker/ibor/jodi/JodiPositionsRepository.java

               ibor-server/src/main/java/com/kmakker/ibor/service/PositionService.java

PHASE 4+ — ENTERPRISE PRODUCTION READINESS
=============================================================

  Data Ingestion Pipeline
  - Kafka topics per asset class (already in original proposal)
  - FTP/SFTP custodian file drops (BNY, State Street)
  - FIX protocol for real-time trade feeds
  - Bloomberg / Refinitiv connector for pricing and reference data

  Reconciliation
  - IBOR vs custodian position breaks (daily)
  - Price validation: stale prices, outliers, missing prices
  - Break reporting and workflow

  Identity & Access Control
  - SSO / SAML (Okta, Azure AD) — non-negotiable for enterprise
  - Role-based access: PM (own portfolio), Risk Officer (all portfolios), Analyst (read-only)
  - Audit trail: who asked what, when (regulatory requirement)

  Compliance & Regulatory
  - IBOR → ABOR reconciliation
  - Exposure / concentration limit monitoring
  - MiFID II / SEC transaction reporting
  - SOC 2 Type II certification (required by most institutional buyers)

  Operational Resilience
  - Disaster recovery (RTO/RPO commitments)
  - Encryption at rest and in transit
  - Multi-region on AWS

  Repo:       ui/ folder inside middleoffice-ibor monorepo (no separate repo yet)

  Layout design (Phase 1 demo):
  - Two-panel: portfolio snapshot (left) + AI chat (right) + data grid (bottom)
  - Portfolio snapshot shows on load: fund name, AUM, as-of date, sector weights,
    asset mix, date range available — enough context for users to form their questions
  - AI introduces itself with capability disclosure, not scripted questions
  - Data grid auto-populates with data relevant to the last AI answer
  - No pre-canned clickable questions (kills credibility)
  - Polite out-of-scope response when user asks beyond available data
  - CORS: gateway must allow http://localhost:5173 (Vite default) + production origin
